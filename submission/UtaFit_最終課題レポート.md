# UtaFit 最終課題レポート

> 2026年 実践のためのWebプログラミング

## 1. テーマ

**自分の安定音域と練習目的から、歌いやすい曲と歌い方を探せる歌唱支援Webアプリ「UtaFit」**を制作した。

カラオケで曲を選ぶとき、知っている曲という理由だけで選ぶと、実際には最高音が高すぎたり、裏声への切り替えやロングトーンが難しかったりする。そこで、曲名だけでなく、音域・声区・難易度・テンポ・歌唱指標から曲を探し、選んだ後の練習までつながるサービスをテーマにした。

## 2. サービスの説明

UtaFitは次の流れで利用する。

1. 曲名、アーティスト、特徴、雰囲気を入力して検索する。
2. 最高音、地声・裏声の最高音、難易度、テンポなど12種類の条件で絞り込む。
3. 自分の安定音域、歌唱レベル、好み、練習目標を入力し、100点満点の相性診断を行う。
4. 曲詳細で音域幅、歌唱指標、難所、似ている曲を確認する。
5. 架空曲「夜明けのキャンバス」では、歌詞上の歌い方注釈と1行練習モードを試す。

収録データは実在曲20曲と、注釈デモ用の架空曲1曲である。実在曲の歌詞は保存せず、音域・BPM・難易度も動作確認用サンプル値であることを画面に明記した。

## 3. 設計の説明

### 3.1 画面構成

- `/`：サービスの価値と最短の入口
- `/songs`：検索、絞り込み、並び替え、0件表示
- `/songs/[id]`：基本情報、歌い方、音域、難所、練習、類似曲
- `/songs/[id]/practice`：1フレーズずつ進める練習モード
- `/recommend`：匿名で使える音域・好み診断
- `/project-story`：制作意図と学びをまとめたブログ記事

### 3.2 データと計算

画面部品、データ、計算処理を分離した。検索は`lib/song-search.ts`、推薦は`lib/recommendation.ts`、歌詞注釈の区間計算は`lib/annotations.ts`にまとめ、UIに依存しない純粋関数としてテストできるようにした。

推薦はブラックボックスのAIではなく、音域35点、難易度20点、ジャンル10点、曲調10点、アーティスト10点、類似曲10点、練習目標5点の合計100点で計算する。点数だけでなく、最大4件の推薦理由も返す。

### 3.3 状態の扱い

検索条件はURLのクエリ文字列に保存した。これにより、再読み込みしても条件が残り、同じ検索結果を共有しやすい。練習の完了状況だけは個人の端末内に閉じる情報として`localStorage`へ保存した。

### 3.4 使用技術

- TypeScript / React / Next.js
- Tailwind CSS
- Vitest
- Node.js
- Git / GitHub（公開時に使用）

## 4. 自分なりにこだわった点

### 推薦理由を説明できるようにした

単に「おすすめ度90点」と表示しても、利用者はなぜ合うのか判断できない。そこで、点数の内訳と、「地声最高音が安定音域内」「好きな雰囲気を含む」などの理由を表示した。生成AIが一般化しても、判断材料を人に返す設計は重要だと考えた。

### 著作権とデータ精度を曖昧にしなかった

実在曲の歌詞を無断で掲載せず、歌詞注釈の体験は架空曲に限定した。また、確認していない音域データを正確な分析結果のように見せず、サンプル値であることを目立つ位置に表示した。機能が動くことと、情報が信頼できることは別問題だからである。

### 重なる歌詞注釈を表現した

同じ文字列に「ミックスボイス」と「ロングトーン」など複数の技法が重なる場合がある。注釈の開始位置と終了位置を境界にして歌詞を分割し、各区間に重なる注釈をまとめる方法を実装した。重複部分は二重線で示し、クリックすると詳細を確認できるようにした。

### MVPの範囲を決めた

アカウント、お気に入り、履歴、データベース接続、AIによる注釈候補生成は将来機能とし、今回は検索・診断・注釈・練習という中心価値へ集中した。作らないものを決めることで、動く状態まで完成させやすくなった。

## 5. 学びになったと感じたこと

- 要件を機能一覧のまま受け取らず、利用者が困る場面に置き換える必要がある。
- URLと画面状態を同期すると、再読み込みや共有に強いWebサービスになる。
- 複雑な表示は、先にデータ構造と計算処理を決めると実装しやすい。
- UIと計算を分離すると、テストで境界値や例外を確認できる。
- AIにコードを書かせる場合でも、仕様、権利、データの正しさは人が判断しなければならない。

自動テストは、検索・推薦・音名変換・歌詞注釈・アクセス判定など19件を作成した。最終確認では全テスト、静的解析、本番ビルドが成功した。

## 6. 感想

最初は曲検索機能を中心に考えていたが、設計を進めるうちに「曲を見つけた後、どう練習するか」までつながらないとサービスとして弱いと感じた。そのため、歌詞注釈と1行練習を追加し、探す・納得する・練習するという流れに整理した。

また、生成AIを使うと実装の速度は上がるが、候補が多く出る分、何を採用し何を削るかを言語化する必要があった。AIの出力を完成品と考えず、設計案を広げる相手として使い、最後はテストと画面で確かめることが大切だと感じた。

## 7. AIの活用方法

AIには、要件整理、データ型と画面構成の案出し、実装のたたき台、テスト観点の洗い出し、文章の推敲を手伝ってもらった。一方で、テーマ、優先順位、MVPの境界、著作権への配慮、推薦を説明可能にする方針は人が判断する項目として扱った。AIが生成した内容は、ソースコード、テスト結果、本番ビルドを確認して採用した。
